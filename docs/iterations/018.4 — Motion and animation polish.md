# Iteration 018.4 ‚Äî Motion and Animation Polish

## Goal

Bring coherence, hierarchy and restraint to animation across the entire Ordboble frontend.

The app already has motion throughout ‚Äî word entrances, loading dots, hover effects, floating particles, success feedback ‚Äî but it was added incrementally. This iteration audits and aligns it into a deliberate system.

Focus on:
- consistent easing vocabulary
- motion hierarchy (primary vs ambient vs micro-interactions)
- theme-aware animation intensity
- `prefers-reduced-motion` respect
- reduced simultaneous animation clutter
- performance-safe properties only

The goal is motion that feels calm, considered and on-brand ‚Äî not an animation demo.

---

# Scope

## In Scope

- Audit and align all easing values in `app.css` and components
- Establish 2‚Äì3 named easing tokens used consistently
- Make `prefers-reduced-motion` work across all keyframe animations and transitions
- Make animation intensity react to `data-theme` attribute (calm = softer/slower, default/playful = current)
- Clean up inline `style="animation: ..."` usages ‚Äî move to utility classes or scoped `<style>`
- Reduce visual noise from concurrent ambient animations on the same screen
- Verify all animated properties use only `transform` and `opacity` where possible
- Keep existing animation character ‚Äî do not flatten everything, refine it

## Out of Scope

Do NOT:
- redesign screens or layouts
- add new animation types or motion concepts
- add physics or spring libraries
- add Svelte `fly/slide` transitions unless replacing something broken
- add new features
- add backend changes
- add routing changes
- add new dependencies unless clearly justified

# Problems to Fix

## 1. No `prefers-reduced-motion` support

The app currently has zero `prefers-reduced-motion` handling. Users who have opted out of motion see all animations regardless. This is both an accessibility issue and a product quality gap.

Fix:
- Add a global `@media (prefers-reduced-motion: reduce)` block in `app.css`
- Disable or instant-play all keyframe animations under this media query
- Reduce CSS transitions to `0.01s` under this media query
- Do NOT remove structural transitions (e.g. opacity state changes) ‚Äî just make them instant

## 2. Inconsistent easing

The codebase uses: `ease`, `ease-out`, `ease-in-out`, `cubic-bezier(0.25, 0.46, 0.45, 0.94)`, `cubic-bezier(0.34, 1.4, 0.64, 1)`, `cubic-bezier(0.34, 1.3, 0.64, 1)`, `cubic-bezier(0, 0, 0.2, 1)`. These were added individually and don't form a vocabulary.

Fix:
- Define 3 named easing tokens in `app.css` `:root`:
  - `--ease-out`: `cubic-bezier(0.25, 0.46, 0.45, 0.94)` ‚Äî standard deceleration, used for most UI motion
  - `--ease-spring`: `cubic-bezier(0.34, 1.3, 0.64, 1)` ‚Äî subtle spring, used for word entrances and pop-ins
  - `--ease-in-out`: `cubic-bezier(0.45, 0, 0.55, 1)` ‚Äî ambient/looping animations
- Update keyframes and utility classes to reference these tokens where possible

## 3. Theme-aware animation intensity is partial

The `[data-theme="calm"]` selector already modifies `word-pop-in`, but nothing else adapts to the theme. On calm themes, ambient pulses, floating particles and entrance animations should also be softer.

Fix:
- Under `[data-theme="calm"]`:
  - Slow down `dot-pulse` duration slightly (1.6s instead of 1.2s)
  - Reduce `pulse-ring` scale and opacity travel
  - Slow down `float-subtle` on FloatingParticle (add `animation-duration` override via data-theme if feasible)
  - Reduce `fade-in` translateY offset from 10px to 4px

- Under `[data-theme="playful"]` (or default):
  - Keep current energy ‚Äî no change needed

## 4. Inline `style="animation: ..."` in component markup

Several components use inline style attributes to set animations, which is harder to override with media queries or theme selectors:
- `ContributorMobileScreen.svelte` line 106: `style="animation: fade-in 0.4s ease forwards;"`
- `PresentationModeScreen.svelte` line 303: `style="animation: fade-in 0.4s ease forwards;"`
- `PresentationModeScreen.svelte` line 344: `style="animation: fade-in 0.8s ease forwards;"`
- `FloatingParticle.svelte`: `style="... animation: float-subtle {duration}s ease-in-out {delay}s infinite"`

Fix:
- Replace `ContributorMobileScreen` and `PresentationModeScreen` inline animation styles with `.animate-fade-in` class (already exists)
- For `PresentationModeScreen` line 344 (0.8s version): add `.animate-fade-in-slow` utility (0.8s variant)
- FloatingParticle keeps dynamic duration via style (it needs a prop-driven value) ‚Äî leave as-is but document

## 5. Simultaneous ambient animation clutter

On the presentation screen, the following can all animate at once:
- Multiple `FloatingParticle` components floating
- `pulse-ring` rings in empty state
- `dot-pulse` loading dots
- `ParticipantCounter` ping dot
- `animate-fade-in` on overlays

This creates visual noise, especially on calm themes.

Fix:
- On calm theme: disable `FloatingParticle` floating animation (`animation: none`) ‚Äî keep the particles visible but static
- On calm theme: reduce `pulse-ring` to single ring instead of two (CSS only, via opacity on second ring)
- Ensure no more than 2 concurrent looping ambient animations are active at any given screen state

## 6. `transition-all` misuse

Several components use `transition-all duration-150/200` which transitions every CSS property and can cause GPU layer promotion for unexpected properties. This is a performance concern especially during word cloud rendering.

Fix:
- Replace `transition-all duration-150` with `transition-[background-color,border-color,color,opacity] duration-150` in editor/picker components
- Keep `transition-soft` (which also uses `all`) only where truly needed ‚Äî flag as intentional

# Visual Direction

Motion after this iteration should feel:

- **Restrained** ‚Äî fewer things moving at once
- **Consistent** ‚Äî same easing vocabulary throughout
- **Theme-aware** ‚Äî calm themes feel calm in motion too
- **Accessible** ‚Äî users who opt out of motion get instant state changes, not animation
- **Performance-safe** ‚Äî only `transform` and `opacity` drive animated properties

Do not:
- make everything feel slower ‚Äî only calm theme should feel slower
- remove animation character ‚Äî word entrances should still feel satisfying
- add motion where there was none unless it clarifies a state change
