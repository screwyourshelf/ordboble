# Iteration 016.4 — Deployed Worker and Production API Wiring

## Goal

Deploy the Cloudflare Worker and wire the deployed frontend to the production API.

The goal is to make the hosted Cloudflare Pages frontend create and use real sessions through the deployed Worker.

This completes the first deployed host flow:

```text
Cloudflare Pages frontend
→ POST production Worker /api/clouds
→ D1
→ /share/:id
→ QR/link to /join/:id
```

---

# Scope

## In Scope

- Prepare Worker for deployment
- Apply D1 migrations to remote database
- Deploy Worker to Cloudflare
- Verify Worker production URL
- Configure Cloudflare Pages environment variable:
  - `VITE_API_BASE_URL`
- Redeploy Pages frontend
- Verify frontend can call deployed Worker
- Verify session creation works on deployed frontend
- Verify `/share/:id` produces correct `/join/:id` URL

## Out of Scope

Do NOT:
- add auth
- add custom domains
- add Durable Objects
- add WebSockets
- add presence
- add rate limiting
- implement export
- redesign UI
- change schema unless required
- add new product features

---

# Required Manual Cloudflare Steps

Some steps must be done manually or with authenticated Wrangler.

## D1 Migration

From:

```text
apps/worker
```

Run:

```text
pnpm exec wrangler d1 migrations apply ordboble-db --remote
```

## Worker Deploy

From:

```text
apps/worker
```

Run:

```text
pnpm exec wrangler deploy
```

## Pages Environment Variable

In Cloudflare Pages project settings, set:

```text
VITE_API_BASE_URL=<deployed-worker-url>
```

Example:

```text
https://ordboble-worker.<account>.workers.dev
```

Then redeploy Pages.

---

# Expected Verification

After deployment:

- `GET <worker-url>/health` returns ok
- deployed Pages app loads
- host can create a session from landing page
- frontend navigates to `/share/:id`
- share page shows a `/join/:id` link
- QR encodes the same `/join/:id` link
