# Iteration 018.2 — Mobile Contributor Polish

## Goal

Refine the contributor mobile experience for real-world live usage.

This iteration focuses on:
- keyboard ergonomics
- touch flow
- submission feel
- viewport behavior
- responsiveness
- tactile polish

The goal is to make the contributor experience feel:
- fast
- reliable
- effortless
- pleasant during live events

especially on phones during workshops, classrooms and presentations.

---

# Scope

## In Scope

- Better mobile spacing and ergonomics
- Improved keyboard behavior
- Safer viewport handling on mobile
- Better submission transitions
- Better success feedback
- Faster-feeling interaction flow
- Improved touch targets
- Reduced accidental scroll/jump issues
- Better input focus handling
- More tactile mobile feel

## Out of Scope

Do NOT:
- redesign contributor screen
- add auth
- add accounts
- add offline mode
- add PWA installation
- add push notifications
- add native wrappers
- add new backend architecture
- add analytics/export

---

# Requirements

## Keyboard Ergonomics

Improve:
- input focus behavior
- keyboard open/close feel
- spacing when keyboard visible
- avoid awkward viewport jumps
- avoid content being hidden behind keyboard

Contributor flow should feel smooth on:
- iPhone Safari
- Android Chrome
- small screens

Avoid:
- large layout shifts
- accidental body scroll
- clipped submit buttons

---

# Submission Flow

Current flow works, but should feel:
- faster
- cleaner
- more tactile

Improve:
- transition timing
- loading feel
- success confirmation feel
- disabled state clarity
- post-submit recovery

Submission should feel:
- lightweight
- satisfying
- responsive

---

# Touch Ergonomics

Improve:
- touch target sizes
- spacing between actions
- thumb reach
- visual press feedback

Buttons should feel intentional on mobile.

---

# Viewport Handling

Handle:
- dynamic mobile browser chrome
- vh/dvh issues
- iOS Safari viewport behavior
- keyboard resize behavior

Avoid:
- bottom cutoff
- jumping layout
- trapped scroll states

Prefer:
- safe viewport sizing
- overflow containment
- stable vertical rhythm

---

# Success Feedback

Success state should:
- feel rewarding
- feel lightweight
- not block flow too long
- encourage another submission naturally

Keep subtle and premium.

---

# Error & Retry Feel

Improve:
- inline errors
- reconnect feel
- retry clarity
- disabled/loading states

Avoid:
- harsh red states
- noisy warnings
- aggressive alerts

---

# Motion Polish

Contributor motion should:
- feel faster than presentation mode
- feel tactile
- use restrained motion

Calm themes:
- softer transitions
- less bounce/glow

Playful themes:
- slightly more energy

Keep motion lightweight.

---

# Technical Constraints

- Svelte 5
- TypeScript
- existing architecture only
- no new dependencies unless clearly justified
- maintain deployed realtime flow
- keep rendering lightweight
- preserve accessibility basics

---

# Expected Files

Likely:
- ContributorMobileScreen.svelte
- WordInputList.svelte
- SuccessConfirmation.svelte
- Button.svelte
- mobile spacing utilities
- app.css

Only modify files required for contributor polish.

# Verification

Verify:
- mobile layout feels stable
- keyboard interactions improved
- submit flow feels faster/smoother
- success transitions improved
- no viewport clipping
- no major layout jumps
- deployed contributor flow still works
- frontend builds/type-checks
