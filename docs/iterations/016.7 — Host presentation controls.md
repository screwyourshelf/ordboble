# Iteration 016.7 — Host Presentation Controls

## Goal

Add lightweight host-side controls for managing the live presentation experience.

This iteration improves the presenter experience during real sessions without changing the core architecture.

The goal is to let the host:
- focus the presentation
- control visual noise
- temporarily stabilize the cloud
- manage submissions during live facilitation

without introducing admin/auth complexity.

---

# Scope

## In Scope

- Fullscreen presentation toggle
- Hide/show presentation chrome
- Freeze/unfreeze cloud layout
- Pause/resume incoming submissions visually
- Clear/reset cloud (frontend-only or backend-backed if trivial)
- Lightweight host control UI
- Keyboard shortcut support where simple
- Maintain deployed realtime flow

## Out of Scope

Do NOT:
- add authentication
- add admin roles
- add permissions
- add moderation tools
- add persistence for host settings
- redesign presentation mode
- add new realtime architecture
- add WebSockets/Durable Objects
- add analytics/export

---

# Requirements

## Fullscreen Toggle

Host should be able to:
- enter fullscreen
- exit fullscreen

Use browser Fullscreen API.

Add:
- visible button
- optional keyboard shortcut (`F` or similar if simple)

Handle:
- fullscreen unsupported gracefully

---

# Presentation Chrome Visibility

Allow host to:
- hide/show presentation chrome

Chrome includes:
- join URL
- join code
- metadata panels
- helper UI

The word cloud itself should remain visible.

Goal:
- cleaner projector/speaker view

State can remain local-only.

---

# Freeze Cloud Layout

Allow host to:
- freeze word positions visually

Current behavior:
- words can reposition/react as new words arrive

When frozen:
- existing words keep current positions
- new incoming words still appear
- no reshuffling/reflow

This should remain frontend-only.

Do not build a complex layout engine.

---

# Pause Incoming Submissions (Visual)

Host should be able to:
- temporarily pause live updates visually

Important:
- submissions should STILL be accepted by backend/D1
- SSE may continue buffering internally

Simplest acceptable behavior:
- ignore/render-hold incoming SSE words while paused
- apply queued words when resumed

Do NOT implement backend submission locking.

---

# Clear / Reset Cloud

Preferred:
- frontend-only reset for presentation session

Acceptable:
- clear only locally visible words
- preserve backend data

Optional if trivial:
- backend clear endpoint

But avoid backend schema changes or admin complexity.

---

# Host Controls UI

Controls should:
- stay lightweight
- fit existing design system
- avoid large control panels
- feel presenter-focused

Acceptable:
- floating control bar
- corner controls
- translucent overlay controls

Do not redesign presentation mode.

---

# Keyboard Shortcuts (Optional)

If simple:

```text
F → fullscreen
C → toggle chrome
P → pause live updates
R → reset cloud
```

Keep implementation minimal.

---

# Technical Constraints

- Svelte 5
- TypeScript
- existing SSE architecture
- no auth/admin system
- no new dependencies unless clearly justified
- no routing rewrite
- no backend redesign

---

# Expected Files

Likely:
- PresentationModeScreen.svelte
- FloatingToolbar.svelte
- WordCluster.svelte
- realtime.ts

Only modify files required for host controls.

---

# Verification

Verify:
- fullscreen works
- chrome hides/shows
- pause/resume works visually
- queued words appear after resume
- reset works
- deployed realtime still functions
- frontend builds/type-checks
- controls do not break mobile view
