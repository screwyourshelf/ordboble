# Iteration 015.3 — API Browser Integration Hardening

## Goal

Make the frontend-to-Worker integration reliable in real browser/deployed scenarios.

This iteration hardens the API/browser boundary after the first live loop has been verified.

Focus on:
- CORS
- EventSource/browser compatibility
- stable word keys
- safe error handling
- deployed frontend compatibility

---

# Scope

## In Scope

- Add CORS handling to Worker API responses
- Add CORS handling for preflight `OPTIONS`
- Ensure SSE endpoint has browser-safe headers
- Improve EventSource error/reconnect behavior if needed
- Use stable IDs as word keys in `WordCluster`
- Verify frontend can call Worker from another origin
- Verify local frontend + local Worker integration still works
- Keep implementation minimal

## Out of Scope

Do NOT:
- add authentication
- add rate limiting
- add presence
- add Durable Objects
- add WebSockets
- add routing
- redesign UI
- change D1 schema unless absolutely necessary
- add new dependencies
- implement production domain setup

---

# Requirements

## CORS

Worker should support browser calls from frontend origins.

For now, use a simple permissive prototype policy:

```text
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST, OPTIONS
Access-Control-Allow-Headers: Content-Type
```

This can be tightened later.

All API JSON responses should include CORS headers.

`OPTIONS` requests should return `204`.

---

# SSE Headers

SSE responses should include:

```text
Content-Type: text/event-stream
Cache-Control: no-cache
Connection: keep-alive
Access-Control-Allow-Origin: *
```

Keep existing heartbeat behavior.

---

# EventSource Behavior

Ensure frontend:
- does not crash on EventSource errors
- logs or handles errors safely
- keeps fallback prototype behavior when no API URL exists
- cleans up subscriptions properly

Do not build complex reconnect logic unless minimal and clearly needed.

---

# WordCluster Keys

Current follow-up:
- `WordCluster` should use stable IDs as keys where available

Update keying so duplicate words from different clients do not collide.

Fallback may use word if no id exists.

# Architect Output

## Plan

Three independent concerns:

**1. CORS (worker)**
- Add `CORS_HEADERS` constant to `apps/worker/src/lib/response.ts`
- Spread into every response helper via `json()`
- Add `preflight()` helper returning `204 No Content` with same headers
- Add `OPTIONS` check at top of router in `apps/worker/src/index.ts`
- Add `Access-Control-Allow-Origin: *` to SSE response in `apps/worker/src/routes/events.ts`

**2. WordCluster stable keys**
- Add optional `id?: string` to `WordEntry` in `apps/web/src/lib/types/word-cloud.ts`
- Change `{#each}` key in `WordCluster.svelte` to `entry.id ?? entry.word`
- Propagate `payload.id` when appending live words in `PresentationModeScreen.svelte`

**3. EventSource error safety**
- In `realtime.ts`, log to `console.error` when `onerror` fires but no handler is provided

**Risk notes:**
- `Access-Control-Allow-Origin: *` is permissive; acceptable for prototype; can be tightened to deployed origin in a future iteration
- `id` on `WordEntry` is optional — no existing code breaks
