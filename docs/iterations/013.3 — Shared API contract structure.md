# Iteration 013.3 — Shared API Contract Structure

## Goal

Establish a shared API contract structure before implementing real backend endpoints.

The goal is to define where request/response contracts, API types and shared validation concepts should live so the frontend and Worker can evolve consistently.

This iteration should create structure only.

No real product API should be implemented yet.

---

# Scope

## In Scope

- Create shared API contract package/folder
- Define basic API response envelope types
- Define shared error response types
- Define placeholder contract files for future:
  - sessions
  - word submissions
  - realtime events
- Make types importable from both frontend and Worker later
- Keep implementation minimal
- Avoid runtime dependencies
- Verify TypeScript compiles where relevant

## Out of Scope

Do NOT:
- implement real API endpoints
- implement session creation
- implement word submission
- implement validation library
- implement D1
- implement SSE
- connect frontend to Worker
- add generated clients
- add OpenAPI
- add complex schema system
- add runtime package dependencies

---

# Architecture Direction

Create a shared package area:

```text
packages/
  shared/
    src/
      api/
        response.ts
        errors.ts
        sessions.ts
        words.ts
        events.ts
      index.ts
    package.json
    tsconfig.json
```

This package is for shared TypeScript contracts only.

No business logic.

---

# Requirements

## API Response Types

Create simple generic types such as:

```ts
export type ApiSuccess<T> = {
  ok: true;
  data: T;
};

export type ApiFailure = {
  ok: false;
  error: ApiError;
};

export type ApiResponse<T> = ApiSuccess<T> | ApiFailure;
```

---

# Error Types

Create explicit error code types.

Initial examples:
- `not_found`
- `validation_error`
- `expired`
- `rate_limited`
- `internal_error`

Keep these simple.
